# Data Model

## System Blueprints (constants, not stored in DB)

Defined as an enum in `lib/constants/system_blueprints.dart`. These are always available to all users.

```dart
enum SystemBlueprint { poop, pee, walk, medication, play, training }
```

Display labels (generated by `buildDisplayLabel()`):
- poop → "pooped"
- pee → "peed"
- walk → "went for a walk"
- medication → "took medication"
- play → "played"
- training → "did training"

For multiple components: "pooped and peed", "pooped, peed, and went for a walk"
Canonical sort order matches enum declaration order.

## ObjectBox Entities

### Pet
```dart
@Entity()
class Pet {
  @Id() int id = 0;
  String name;
  String? species;
  String? breed;
  DateTime? birthdate;
  String? photoPath;       // local file path; swaps to URL when Firebase added

  @Backlink('pet')
  final events = ToMany<Event>();

  @Backlink('pet')
  final customBlueprints = ToMany<CustomBlueprint>();
}
```

### CustomBlueprint
User-defined per pet. Components restricted to system blueprints only for now (no nested custom blueprints).

```dart
@Entity()
class CustomBlueprint {
  @Id() int id = 0;
  String name;
  String? iconKey;
  // JSON string: [{"type":"system","key":"poop"}, ...]
  // Restricted to system blueprints only in PoC
  String componentsJson;

  final pet = ToOne<Pet>();
}
```

### Event
One record per pet per logging action. If a user logs poop+pee for Sarge and Winnie, two Event records are written.

```dart
@Entity()
class Event {
  @Id() int id = 0;

  // System blueprint keys included e.g. ["poop", "pee"]
  // ObjectBox supports List<String> natively
  List<String> systemComponents;

  // IDs of any CustomBlueprints included
  List<int> customBlueprintIds;

  DateTime timestamp;
  String loggedByName;    // plain name string for PoC; becomes uid when Firebase added

  // Generated at write time, never recomputed at render
  // e.g. "pooped and peed", "went for a walk and played"
  String displayLabel;

  final pet = ToOne<Pet>();
}
```

---

## Relationships
- One Pet → many Events (backlink)
- One Pet → many CustomBlueprints (backlink)
- One Event → one Pet (ToOne)
- One CustomBlueprint → one Pet (ToOne)

---

## "Last Parent" Rule (for future multi-user)
At least one user with role "parent" must exist per pet. Enforced in app logic before writes, not in DB constraints. Before a parent removes themselves, check count of parents — if 1, block and prompt to promote another user or delete the pet.
